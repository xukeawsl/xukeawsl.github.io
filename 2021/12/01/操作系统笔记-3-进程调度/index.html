操作系统笔记(3)-进程调度

简介

  • 在操作系统获取 CPU 控制权之后, 可以选择调度执行的进程以提高工作效率

  • 为了实现这个目的, 出现了许多种调度策略, 它们各有优势, 是操作系统发展中不可或缺的一部分

工作负载假设

我们对操作系统中运行的进程（有时也叫工作任务）作出如下假设：

  • 每一个工作运行相同的时间

  • 所有工作 同时 到达

  • 一旦开始, 每个工作保持运行 直到完成

  • 所有工作只是用 CPU （即它们不执行 I/O 操作）

  • 每个工作的运行时间是 已知 的

调度指标

  • 周转时间（turnaround time）

    定义为任务完成时间减去任务到达系统的时间，即 $T_{周转时间} = T_{完成时间} - T_{到达时间}$

    因为假设所有工作同时到达, 故 $T_{到达时间} = 0$ , $T_{周转时间} = T_{完成时间}$

先进先出 (FIFO)

  • 先进先出（First In First Out）也叫先进先服务, 意思就是任务按到达系统的先后执行, 假如有三个长度为 100 的任务, 则按照 FIFO 的方式, 其 $T_{平均周转时间} = (100 + 200 + 300) / 3 = 200$

  • 它的缺点也很明显, 就是当任务的长度不都一致时, 周转时间首任务到达的顺序影响较大, 如果时间占用长的任务先执行, 就会导致平均周转时间大大 增加 , 这种问题也被称为 护航效应 , 因此有了后面的 SJF 调度

最短任务优先 (SJF)

  • 最短任务优先（Shortest Job First）即按工作的长度从小到大执行, 优先完成短的任务, 这样可以显著 降低 平均周转时间

  • 缺点是, 假设我们的任务不是同时到达的, 那么假如一个长任务先到达并开始执行, 那么中途加入的短任务就要等到长任务执行完之后再运行, 增加 了平均周转时间

  • 为了优化上述问题, 提出了 STCF 调度策略

最短完成时间优先 (STCF)

  • 为了实现这个策略，我们需要放宽假设, 允许任务未完成时切换, 向 SJF 添加抢占, 称为最短完成时间优先（Shortest Time-to-Completion First）或抢占式作业优先（Preemptive Shortest Job First）调度程序

  • STCF 每当新任务进入系统时, 它就会确定剩余工作和新工作中, 谁的 剩余时间最少 , 然后调度该工作, 这样可以 减少 平均周转时间

新度量指标

  • 响应时间（response time）

    考虑到现代系统对 用户交互 的关注, 引出了 响应时间 这一新的度量指标，它被定义为从任务到达系统到首次运行的时间，即 $T_{响应时间} = T_{首次运行} - T_{到达系统}$

轮转 (RR)

  • 轮转调度（Round-Robin）是为了提高响应时间而提出的一种进程调度策略, 它的基本思想就是在一个 时间片 内运行一个进程, 然后切换到另一个进程执行, 其中 时间片 必须是 时钟中断周期 的整数倍, 因为每隔一个 时钟中断周期 操作系统才能获取 CPU 的控制权, 因此为了正确切断, 必须满足这个条件

  • 使用这种方式，按理来说只要 时间片 设置得越小, 系统的平均响应时间就会越快，但是随之而来的是大量的 上下文切换 , 如果 时间片 过小, 上下文切换 的成本就会很大，因此需要确定一个合理的 时间片 来摊销 上下文切换 的成本

  • 并且采用这种方式，平均周转时间可能比 FIFO 的方式还要差，因为任务的完成时间基本上都被延后了

结合 I/O

  • 考虑到实际情况, 继续放宽假设，任务可能执行 I/O 操作，因此为了更好的利用 CPU ，操作系统显然要在工作发起 I/O 时作出决定，因为当前正在运行的作业在 I/O 期间不会使用 CPU ，它会等待 I/O 完成

  • 在等待 I/O 的过程中，操作系统就可以切换到其它的任务上执行，以提高 利用率

多级反馈队列 (MLFQ)

  • 多级反馈队列（Multi-level Feedback Queue）需要解决两个方面的问题，一是优化 周转时间，通过 SJF 可知这需要先执行短任务，但是实际系统并不能预先知道每个任务执行的时间, 二是为了提高用户的交互体验，需要降低 响应时间

  • 为了解决这个问题，MLFQ 采用多个独立的优先级队列来进行任务调度，基本规则是高优先的队列中的任务先执行，然后同优先级的队列中的任务采用 轮转 的调度方式

基本规则

  1. 如果 A 的优先级 > B 的优先级（不运行B）
  2. 如果 A 的优先级 = B 的优先级，轮转运行
  • 缺点，优先级低的任务迟迟得不到 CPU 来运行

如何改变优先级

由于我们一开始无法确定任务的类型，到底是运行时间很短、频繁放弃 CPU 的交互型工作还是需要很多 CPU 时间、响应时间却不重要的长时间计算密集型工作，故假设每个任务一开始都是最高优先级，每运行一个时间片就降低其优先级，因为耗时短的任务很快就会完成

  1. 工作进入系统时，放在最高优先级（最上层队列）
  2. 工作用完整个时间片后，降低其优先级（移入下一个队列）
  3. 如果工作在其时间片以内主动放弃 CPU，则优先级不变

这样，对单个长任务来说，要不了多久就会到达最低优先级的队列，短任务则很快就能执行完，如果有频繁的 I/O 操作，它会在时间片用完之前放弃 CPU ，这种情况下它的优先级不变

  • 缺点，如果有太多的交互性工作（或恶意任务），就会不断占用 CPU，导致长任务得不到 CPU

提升优先级

为了解决长任务得不到运行的情况，并且当一个 CPU 密集型工作变成了交互型时能够更好的得到 CPU 。提出了一种方式解决，即每隔一段时间就提升所有任务的优先级到最高级，这样就能比较好地解决以上问题

  1. 经过一段时间 S，就将系统中所有工作重新加入最高优先级队列中
  • 缺点，S 的大小很难确定，如果设置得太高，长工作会饥饿，如果设置得太低，交互型工作又得不到合适的 CPU 比例，因为近似于全部任务都是轮转的

更好的计时方式

为了防止恶意任务的出现（在时间片用完之前调用 I/O 操作，以保持高优先级，占用大量 CPU 资源），将规则 4 和 5 进行更改，即

  1. 一旦工作用完了其在某一层中的时间配额（无论中间主动放弃了多少次 CPU），就降低其优先级（移入低一级队列）

有了这样的保护后，不论进程的 I/O 情况如何，都会慢慢地降低优先级，有效降低了恶意任务的危害

MLFQ 小结

根据一系列的优化，得到了如下的规则：

  1. 如果 A 的优先级 > B 的优先级（不运行B）
  2. 如果 A 的优先级 = B 的优先级，轮转运行
  3. 工作进入系统时，放在最高优先级（最上层队列）
  4. 一旦工作用完了其在某一层中的时间配额（无论中间主动放弃了多少次 CPU），就降低其优先级（移入低一级队列）
  5. 经过一段时间 S，就将系统中所有工作重新加入最高优先级队列中

练习：C 语言模拟 MLFQ

接口设计

为了方便实现, 实现不考虑 I/O 的情况, 并且所有任务都在同一时刻 0 加入队列中, 接口设计如下：

  • initMLFQ(unsigned int count, unsigned int slice) : 初始化 MLFQ, 队列数为 count, 所有队列中轮转的时间片为 slice

  • addJob(MLFQ* mlfq, unsigned int id, unsigned int cost) : 向 MLFQ 添加工作, 需要提供工作 id 和花费 cost

  • dowork(MLFQ* mlfq, unsigned int uptime) : 开始工作, uptime 表示每隔多少时间提升所有任务的优先级

// 头文件 MLFQ.h 定义如下
#ifndef __MLFQ__H__
#define __MLFQ__H__

#include <stdio.h>
#include <stdlib.h>
#include <stdbool.h>

// 任务结点
typedef struct Job {
    unsigned int id;             // 任务 id
    unsigned int cost;           // 任务需要花费的时间(ms)
    struct Job* next;                   // 指向下一个 Job
} Job, THead;

// MLFQ 结构体
typedef struct MLFQ {
    unsigned int count;                 // 任务队列的数量
    unsigned int slice;                 // 每个队列时间片长度
    THead** lst;                        // 多级队列指针
} MLFQ;


// 初始化 MLFQ
MLFQ* initMLFQ(unsigned int count_, unsigned int slice_) {
    MLFQ* mlfq = (MLFQ*)malloc(sizeof(MLFQ));
    mlfq->count = count_;
    mlfq->slice = slice_;
    mlfq->lst = (THead**)calloc(mlfq->count, sizeof(THead*));
    for (int i = 0; i < mlfq->count; i++) {
        mlfq->lst[i] = (THead*)malloc(sizeof(THead));
        mlfq->lst[i]->next = NULL;
    }
    return mlfq;
}

// 向 MLFQ 添加任务
void addJob(MLFQ* mlfq, unsigned id_, unsigned int cost_) {
    Job* job = (Job*)malloc(sizeof(Job));
    job->id = id_;
    job->cost = cost_;
    job->next = mlfq->lst[0]->next;
    mlfq->lst[0]->next = job;
}

// 将所有任务都提升到最高优先级
void upAllLevel(MLFQ* mlfq) {
    for (int i = 1; i < mlfq->count; i++) {
        while (mlfq->lst[i]->next) {
            Job* job = mlfq->lst[i]->next;
            mlfq->lst[i]->next = job->next;
            job->next = mlfq->lst[0]->next;
            mlfq->lst[0]->next = job;
        }
    }
}

// 释放内存
void delMLFQ(MLFQ* mlfq) {
    for (int i = 0; i < mlfq->count; i++) {
        Job *pre = mlfq->lst[i], *cur = pre->next;
        while (cur) {
            Job* tmp = cur;
            pre->next = cur->next;
            cur = pre->next;
            free(tmp);
        }
        free(mlfq->lst[i]);
    }
    free(mlfq);
}


// 开始工作
void doWork(MLFQ* mlfq, unsigned int uptime) {
    unsigned int tick = 1;
    for (int i = 0; i < mlfq->count; i++) {
        Job *pre = mlfq->lst[i], *cur = pre->next;
        while (cur) {

            if (tick % uptime == 0) {   // 间隔了 uptime 时间
                printf("tick [%u] upAllLevel\n", tick);
                upAllLevel(mlfq);
                tick++;
                i = -1;
                break;
            }

            printf("tick [%u] run Job [%u] on Level [%d]\n", tick, cur->id, mlfq->count - i);

            if (!(--cur->cost)) {   // 任务完成
                printf("Job [%u] complete!", cur->id);
                Job* tmp = cur;
                pre->next = cur->next;
                cur = pre->next;
                free(tmp);
            }
            else {
                if (tick % mlfq->slice == 0) {  
                    if (i == mlfq->count - 1) { // 已经是最低优先级了
                        pre = pre->next;
                        cur = cur->next;
                    }
                    else {  // 降级当前任务
                        pre->next = cur->next;
                        cur->next = mlfq->lst[i + 1]->next;
                        mlfq->lst[i + 1]->next = cur;
                        cur = pre->next;
                    }
                }
            }

            tick++;
        }

        // 检查当前队列是否还有任务
        if (i != -1 && mlfq->lst[i]->next) {
            i--;    // 有的话下次继续
        }
    }

    delMLFQ(mlfq);
}

#endif

测试

#include "./MLFQ.h"

int main(int argc, char* argv[]) {
    // 创建三级反馈队列, 每个队列的轮转时间片是 10ms
    MLFQ* mlfq = initMLFQ(3, 10);

    // 加入任务, 假设同时刻加入
    addJob(mlfq, 0, 80);        // id : 0, cost : 80
    addJob(mlfq, 1, 100);       // id : 1, cost : 100
    addJob(mlfq, 2, 20);        // id : 2, cost : 20

    doWork(mlfq, 100);   // 每隔 50 ms 将全部任务提升到最高优先级

    return 0;
}

设计缺陷

  • 由于在改变优先级时采用的 头插法, 故有失公平性, 若频繁更新优先级, 可能导致一个任务重复占用时间片

  • 接口设计没有作输入检查, 假设用户的输入都是合理的

拓展: 比例份额

  • 比例份额（proportional-share）调度的最终目标，是确保每个工作都获得一定比例的 CPU 时间，而不是优化 周转时间 和 响应时间

  • 与其相关的调度算法主要有两种：彩票调度 和 步长调度

彩票调度 (Lottery Scheduling)

  • 用彩票数（ticket）代表进程占有某个资源的份额，比如 A 进程和 B 进程分别有 75 和 25 张彩票，那么我们希望进程 A 占用 75% 的 CPU 时间而进程 B 占用 25% 的 CPU 时间

  • 通过随机 0 ~ 99 中的数来决定执行哪个进程

C 语言模拟彩票调度

#include <stdio.h>
#include <stdlib.h>
#include <time.h>

unsigned int totaltickets;

typedef struct Job {
    unsigned int id;            // 任务 id
    unsigned int times;         // 任务需要执行几次 cpu 时间片
    unsigned int tickets;       // 任务票数
    struct Job *next;           // 连接下一个任务
} Job, Lott;

Lott* initLott() {
    srand(time(NULL));          // 设置随机种子
    totaltickets = 0;
    Lott* lott = (Lott*)malloc(sizeof(Lott));
    lott->next = NULL;
    return lott;
}

void addJob(Lott* lott, unsigned int id_, unsigned int times_, unsigned int tickets_) {
    Job* job = (Job*)malloc(sizeof(Job));
    job->id = id_;
    job->times = times_;
    job->tickets = tickets_;
    job->next = lott->next;
    lott->next = job;
    totaltickets += job->tickets;
}

void doWork(Lott* lott) {
    int tick = 0;   // 记录时钟
    while (lott->next) {
        int winner = rand() % totaltickets;
        int counter = 0;
        Job *pre = lott, *cur = pre->next;
        while (cur) {
            counter += cur->tickets;
            if (counter > winner) {     // 找到了需要执行的任务
                ++tick;
                printf("winner is %d => ", winner);
                printf("tick [%d] : run Job [%u] times [%u] (totaltickets : %u)\n",
                        tick, cur->id, cur->times, totaltickets);
                
                if (!(--cur->times)) {
                    printf("Done: Job [%u] compelte !\n", cur->id);
                    totaltickets -= cur->tickets;
                    pre->next = cur->next;
                    free(cur);
                    cur = pre->next;
                }

                break;
            }

            pre = pre->next;
            cur = cur->next;
        }
    }

    free(lott);
}

int main(int argc, char* argv[]) {

    Lott* lott = initLott();

    /*
        一个可以优化的地方就是按票数从大到小排序后再加入任务队列, 可以更快
        得遍历任务
    */
    addJob(lott, 0, 5, 25);
    addJob(lott, 1, 3, 25);
    addJob(lott, 2, 8, 50);

    doWork(lott);

    return 0;
}

缺点 : 彩票调度 算法只有当工作执行非常多的时间片时，才能得到期望的结果, 因此提出了 步长调度 算法来改进

步长调度 (Stride Scheduling)

  • 这种算法引入了步长的概念，它与彩票数成反比，每次进程运行后我们会让它的计数器增加它的步长，记录它的总体进展，每次从任务队列中选一个步长最短的出来执行

C 语言模拟步长调度

#include <stdio.h>
#include <stdlib.h>

typedef struct Stride {
    unsigned int id;    // 任务 id
    unsigned int pass;  // 行程值
    unsigned int len;   // 步长
    unsigned int times; // 需要执行从 cpu 时间片次数
    struct Stride *next;
} Stride;

Stride* initStride() {
    Stride* stride = (Stride*)malloc(sizeof(Stride));
    stride->next = NULL;
    return stride;
}

void addJob(Stride* stride, unsigned int id_, unsigned int times_, unsigned int len_) {
    Stride* job = (Stride*)malloc(sizeof(Stride));
    job->id = id_;
    job->pass = 0;  // 行程值一开始都是0
    job->len = len_;
    job->times = times_;
    job->next = stride->next;
    stride->next = job;
}

void insert(Stride* stride, Stride* job) {
    Stride *pre = stride, *cur = pre->next;

    while (cur) {
        if (cur->pass >= job->pass) {
            job->next = cur;
            pre->next = job;
            return;
        }
        pre = pre->next;
        cur = cur->next;
    }

    // cur 不存在
    pre->next = job;
    job->next = NULL;
}

void doWork(Stride* stride) {
    int tick = 0;

    while (stride->next) {
        // 取出第一个节点
        Stride* first = stride->next;
        first->pass += first->len;      // 行程值加上步长
        printf("tick [%d] : Job [%u] run , pass [%u], times[%u], len[%u]\n",
                ++tick, first->id, first->pass, first->times, first->len);
        
        if (!(--first->times)) {
            stride->next = first->next;
            printf("Done: Job [%u] compelete!\n", first->id);
            free(first);
        }
        else {
            stride->next = first->next;
            insert(stride, first);
        }
    }

    free(stride);
}


int main(int argc, char* argv[]) {
    
    Stride* stride = initStride();

    addJob(stride, 0, 4, 100);
    addJob(stride, 1, 3, 50);
    addJob(stride, 2, 5, 250);
    
    doWork(stride);

    return 0;
}

缺点 : 步长调度 的缺点就是当新任务加入时，如何设置行程值很困难，并且需要对每个进程记录全局状态，而 彩票调度 加入一个新任务很简单，只需要更新全局票数然后插入节点即可

小结

  • 这两种调度策略都没有被广泛采用，因为确定票数是很困难的，并且它们不能很好地结合 I/O , 但是在一些特定的应用场景还是很有用的
